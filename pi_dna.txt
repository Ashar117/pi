PHASE 1: FIRST REBUILD ATTEMPT (Week 4)
---------------------------------------
Attempted Traditional MVC Architecture:

Structure Created:
├── nlu/
│   └── intent_parser.py        (regex patterns)
├── execution/
│   └── execution_engine.py     (if/else chains)
├── memory/
│   ├── memory_manager.py       (complex state machine)
│   ├── sqlite_store.py
│   ├── supabase_store.py
│   └── l3_builder.py
├── integrations/
│   ├── gmail_client.py
│   ├── weather.py
│   └── calendar.py
└── main_v2.py

Architecture:
User Input → Intent Parser → Execution Engine → Memory Manager → Response

Problems Encountered:
1. Hard-coded patterns couldn't handle variations
   - "remember X" worked
   - "don't forget X" failed
   - "keep in mind X" failed
   
2. Intent classification failures
   - "what's the weather" → calendar intent (wrong)
   - "schedule meeting" → weather intent (wrong)
   
3. Memory manager was over-engineered
   - 3 separate stores (L1/L2/L3)
   - Complex sync logic
   - Race conditions
   - Data consistency issues
   
4. Import errors everywhere
   - Circular dependencies
   - Missing __init__.py files
   - Wrong relative imports
   
5. Files imported deleted modules
   - app/main.py imported memory.sqlite_store (deleted)
   - core/morning_briefing.py imported integrations.weather (deleted)
   
6. Hardcoded paths everywhere
   - "E:/pi/data/pi.db" (breaks on other machines)
   - "E:/pi/prompts/system.txt" (not portable)

Critical Bugs Found:
- `results` variable undefined in memory_manager.py line 96
- `supabase_client` vs `client` naming mismatch
- `get_memory_by_source()` function didn't exist
- Column names wrong (code used `memory_type`, schema had `tier`)
- Bare `except:` blocks hiding real errors
- Deprecated datetime.utcnow() calls
- Claude API rejected system-role messages in messages array

File Count: ~80 files, ~3000+ lines of code
Result: COMPLETE FAILURE - wouldn't run, broken architecture

Key Insight from Ash: "Stop building an app. Build an agent."

TRIAL & ERROR LESSONS:
----------------------
1. LESSON: Don't build complex before simple
   - Started with 3-tier memory before getting chat working
   - Should have: Basic chat → Memory → Tools → Features
   
2. LESSON: Intelligence doesn't belong in code
   - Spent weeks writing regex patterns
   - Claude can understand variations naturally
   
3. LESSON: Over-engineering kills projects
   - memory_manager.py: 400+ lines for what tools_memory.py does in 300
   - Complexity = bugs = failure
   
4. LESSON: Modular means replaceable
   - Can swap Groq for OpenAI easily
   - Can add new tools without touching core
   - Can modify consciousness without rewriting code
   
5. LESSON: Test incrementally
   - Building everything then testing = disaster
   - Build one feature → test → next feature
   
6. LESSON: Hard-coded paths are evil
   - Broke when running from different directory
   - Broke when sharing with others
   - Always use os.path with __file__
   
7. LESSON: Errors must be visible
   - Bare except: blocks hid real problems
   - Always log exceptions with context
   
8. LESSON: Database schema != Code schema
   - Column name mismatches caused silent failures
   - Schema first, code follows schema

tools/tools_memory.py (380 lines)
---------------------------------
Purpose: Memory read/write/delete operations, L3 context management
Dependencies: supabase, sqlite3, json, datetime

Class: MemoryTools
Methods:
- __init__(supabase_url, supabase_key, sqlite_path=None)
- _init_sqlite(): Create l3_cache table
- memory_read(query, tier=None): Search L3/L2
- memory_write(content, tier, importance, category, expiry): Store
- memory_delete(target, soft=True): Remove/archive
- get_l3_context(max_tokens=800): Format for prompt injection
- _sync_l3(): Pull Supabase → SQLite
- _verify_write(entry_id, content, tier): Verify storage
- _generate_id(): Create unique IDs

Storage Strategy:
- L3: Dual-write (Supabase + SQLite cache)
- L2: Supabase only (large, searchable)
- L1: Supabase only (archive)
- Reads from SQLite first (fast), fallback to Supabase

5. MEMORY ARCHITECTURE (3-Tier System)
================================================================================

PHILOSOPHY:
----------
Human memory doesn't store everything equally. Neither should Pi.

- L3: What you're actively thinking about (working memory)
- L2: What you know and can recall when needed (long-term memory)
- L1: Everything that ever happened (episodic memory)

THREE-TIER DESIGN:

┌────────────────────────────────────────────────────────────────┐
│ L3: ACTIVE CONTEXT (Always Loaded)                             │
├────────────────────────────────────────────────────────────────┤
│ Storage:   Supabase (cloud) + SQLite (local cache)             │
│ Size:      ~800 tokens max (~3000 chars)                       │
│ Loaded:    On EVERY agent startup                              │
│ Purpose:   Current state, active projects, pending reminders   │
│ Lifetime:  Until expired or manually demoted                   │
│ Access:    Instant (SQLite cache)                              │
│                                                                 │
│ Examples:                                                       │
│ - "Working on GNN paper until March 15" (expires: 2026-03-15)  │
│ - "Meeting tomorrow at 2pm" (expires: tomorrow+1 day)          │
│ - "Ash's permanent profile" (never expires)                    │
│ - "Focus on cost optimization this week" (expires: Sunday)     │
│                                                                 │
│ Criteria for L3:                                                │
│ - Importance ≥ 8/10                                             │
│ - Time-sensitive (has expiry date)                             │
│ - Needed for every session                                     │
│ - Permanent profile data                                       │

┌────────────────────────────────────────────────────────────────┐

│ Size:      Unlimited (but compressed)                          │

│ Purpose:   Structured long-term knowledge                      │

│ - "What did we discuss last Tuesday?"                          │
│ - "Show me all tool failures this week"                        │
│ - "When did I start working on project X?"                     │

│ Archival:                                                       │
│ After 30 days → compress to local JSONL                        │
│ After 90 days → delete from Supabase                           │
│ Local archives kept indefinitely (disk is cheap)               │

MEMORY FLOW:

User tells Pi something
    │
    ▼
Claude analyzes importance
    │
    ├─ Importance 1-3 → Write to L1 only (log it)
    │
    ├─ Importance 4-7 → Write to L2 (organized knowledge)
    │
    └─ Importance 8-10 → Write to L3 (active context)
                          ↓
                     Set expiry date?
                          ├─ Yes → Auto-demote to L2 after expiry
                          └─ No → Stays in L3 until manually removed

L3 SYNC STRATEGY:

Write Operation:
1. Write to Supabase (cloud, authoritative)
2. Write to SQLite (local, cache)
3. Verify both succeeded
4. Return success to Claude

Read Operation:
1. Read from SQLite (instant)
2. If data seems stale (check sync timestamp)
   → Pull from Supabase to refresh cache
3. Return data to Claude

Startup:
1. Pull all L3 from Supabase
2. Overwrite entire SQLite l3_cache table
3. Mark sync timestamp
4. Load into context string

DATA PERSISTENCE GUARANTEES:

L3:
- Dual-write (Supabase + SQLite) = redundancy
- Supabase is authoritative source
- SQLite can be rebuilt from Supabase
- Synced every startup

L2:
- Supabase only
- No local cache (size unlimited)
- Backed up by Supabase automatic backups

L1:
- Supabase for 30 days
- Local JSONL archive after 30 days
- Multiple archive copies recommended

Disaster Recovery:
1. If SQLite corrupted → rebuild from Supabase L3
2. If Supabase down → use SQLite cache (read-only mode)
3. If both lost → restore from backup (manual process)

6. TOOL SYSTEM SPECIFICATIONS
================================================================================

TOOL PHILOSOPHY:
---------------
Tools are pure executors. They don't make decisions. They do what they're told.

Intelligence = Claude (consciousness.txt)
Tools = Execute actions
Verification = Claude sees results, decides next action

SEVEN CORE TOOLS:

1. memory_read
2. memory_write
3. memory_delete
4. execute_python
5. execute_bash
6. modify_file
7. read_file

(create_file and list_files are also available but used less frequently)

TOOL 1: memory_read
-------------------
Purpose: Search memory across tiers
Returns: List of matching entries

Parameters:
- query (required): Search string
- tier (optional): "l1", "l2", "l3", or None (search all)

Behavior:
- If tier="l3": Search SQLite l3_cache only (fast)
- If tier="l2": Search Supabase organized_memory (comprehensive)
- If tier=None: Search both L3 + L2, merge results

Return Format:
```json
[
    {
        "id": "abc-123",
        "content": "Working on GNN paper",
        "importance": 8,
        "category": "active_project",
        "tier": "l3",
        "active_until": "2026-03-15"
    },
    {
        "id": "def-456",
        "title": "GNN Paper Outline",
        "content": {"sections": ["intro", "methods", "results"]},
        "importance": 6,
        "tier": "l2"
    }
]
```

Usage Example:
```python
# Claude sees user message: "what am I working on?"

# Claude calls:
memory_read(query="working on", tier="l3")

# Tool returns:
[
    {
        "content": "Working on GNN paper until March 15",
        "importance": 8,
        "tier": "l3"
    }
]

# Claude responds:
"You're working on a GNN paper, deadline March 15."
```

TOOL 2: memory_write
--------------------
Purpose: Store information in memory
Returns: Success confirmation + entry ID

Parameters:
- content (required): What to store
- tier (required): "l1", "l2", or "l3"
- importance (required): 1-10
- category (required): Classification
- expiry (optional): ISO date string

Behavior:
1. Generate unique ID
2. Write to Supabase table (l3_active_memory or organized_memory)
3. If tier="l3": Also write to SQLite cache
4. Verify write succeeded
5. Return confirmation

Return Format:
```json
{
    "success": true,
    "entry_id": "abc-123",
    "tier": "l3",
    "verified": true
}
```

Verification Process:
1. After write, immediately read back from storage
2. Compare content hash
3. If match → verified=true
4. If mismatch → verified=false, return error

Usage Example:
```python
# User: "remember I'm switching to RL research"

# Claude calls:
memory_write(
    content="Research focus changed from GNN to Reinforcement Learning (Apr 2026)",
    tier="l3",
    importance=10,
    category="permanent_profile",
    expiry=None
)

# Tool executes:
1. Generates ID: "xyz-789"
2. Writes to Supabase l3_active_memory
3. Writes to SQLite l3_cache
4. Reads back from both
5. Verifies content matches

# Tool returns:
{
    "success": true,
    "entry_id": "xyz-789",
    "verified": true
}

# Claude sees success → responds:
"Updated permanent profile - research focus now RL."
```

TOOL 3: memory_delete
---------------------
Purpose: Remove or archive memory entries
Returns: Deletion confirmation

Parameters:
- target (required): Entry ID or search query
- soft (optional): True=archive, False=permanent delete (default: True)

Behavior:
- Soft delete: Move to L2 with status="archived"
- Hard delete: Permanently remove from all storage
- Log deletion to L1 audit trail

Return Format:
```json
{
    "success": true,
    "deleted": "abc-123",
    "type": "soft",
    "archived_to": "l2"
}
```

Safety:
- Soft delete is default
- Hard delete requires soft=False explicitly
- Permanent profile items require confirmation
- All deletions logged

TOOL 4: execute_python
----------------------
Purpose: Run Python code
Returns: stdout, stderr, exit code

Parameters:
- code (required): Python code as string

Behavior:
1. Write code to temporary file in /tmp/
2. Execute: `python /tmp/temp_XXXXX.py`
3. Capture stdout + stderr
4. 30 second timeout
5. Return output

Return Format:
```json
{
    "success": true,
    "output": "1048576\n",
    "error": "",
    "code": 0
}
```

Security:
- No sudo/su allowed
- Timeout prevents infinite loops
- Temp files auto-deleted
- No network access (future: sandbox)

Usage Example:
```python
# User: "calculate 2^20"

# Claude calls:
execute_python(code="print(2**20)")

# Tool executes:
Creates /tmp/temp_abc123.py:
    print(2**20)
Runs: python /tmp/temp_abc123.py
Captures: "1048576\n"

# Tool returns:
{
    "success": true,
    "output": "1048576\n",
    "code": 0
}

# Claude responds:
"2^20 = 1,048,576"
```

TOOL 5: execute_bash
--------------------
Purpose: Run bash commands
Returns: stdout, stderr, exit code

Parameters:
- command (required): Bash command
- cwd (optional): Working directory

Behavior:
1. subprocess.run(command, shell=True, cwd=cwd)
2. 30 second timeout
3. Capture all output

Return Format:
```json
{
    "success": true,
    "output": "total 20\n-rw-r--r-- 1 user...",
    "error": "",
    "code": 0
}
```

Security:
- Blacklist: sudo, su, rm -rf /
- Timeout: 30 seconds
- cwd restricted to project_root and subdirs

TOOL 6: modify_file
-------------------
Purpose: Edit files with find-replace
Returns: Success + verification

Parameters:
- path (required): File path
- old_str (required): Exact string to find
- new_str (required): Replacement string

Behavior:
1. Read file
2. Check old_str appears exactly once
3. Replace old_str with new_str
4. Write file
5. Read back and verify

Return Format:
```json
{
    "success": true,
    "verified": true,
    "changes": 1
}
```

Safety:
- old_str must appear exactly once (no ambiguity)
- Verification reads file back
- If verification fails, file is reverted
- Backup created automatically

Usage Example:
```python
# Claude wants to fix a bug

# Claude calls:
modify_file(
    path="E:/pi/pi_agent.py",
    old_str='model="claude-sonnet-4-20250514"',
    new_str='model="claude-sonnet-4-6"'
)

# Tool executes:
1. Reads pi_agent.py
2. Finds 2 occurrences → ERROR (must be unique)

# Tool returns:
{
    "success": false,
    "error": "old_str found 2 times, must be unique"
}

# Claude sees error → tries more specific match:
modify_file(
    path="E:/pi/pi_agent.py",
    old_str='message = self.claude.messages.create(\n            model="claude-sonnet-4-20250514"',
    new_str='message = self.claude.messages.create(\n            model="claude-sonnet-4-6"'
)

# Now unique → succeeds
```

TOOL 7: read_file
-----------------
Purpose: Read file contents
Returns: File contents as string

Parameters:
- path (required): File path
- lines (optional): [start, end] line range

Behavior:
- If lines=None: Read entire file
- If lines=[10, 20]: Read lines 10-20
- If lines=[10, -1]: Read from line 10 to end

Return Format:
```json
{
    "success": true,
    "content": "import sys\nimport os\n...",
    "lines_read": [1, 50]
}
```

def analyze_costs(days=7):
    interactions = get_last_n_days(days)
    
    costs_by_mode = {
        "normie": 0.0,
        "root": 0.0,
        "research": 0.0
    }
    
    for interaction in interactions:
        mode = interaction["mode"]
        cost = interaction["cost"]
        costs_by_mode[mode] += cost
    
    total_cost = sum(costs_by_mode.values())
    avg_cost_per_message = total_cost / len(interactions)
    
    return {
        "total": total_cost,
        "by_mode": costs_by_mode,
        "average": avg_cost_per_message,
        "projection_monthly": total_cost * (30 / days)
    }

def identify_improvements(self):
    improvements = []

    tool_stats = self.analyze_performance(days=30)
    
    for tool_name, stats in tool_stats.items():
        if stats["success_rate"] < 0.80:

    mode_usage = self._get_mode_usage(days=30)

            f"Root mode used {root_percentage:.1%} of the time. "
            "Consider if more tasks could use normie mode to reduce costs."
        )
    
    # Check cost trends
    costs = self.analyze_costs(days=30)
    if costs["projection_monthly"] > 10.0:

            f"Projected monthly cost: ${costs['projection_monthly']:.2f} "

    patterns = self._get_all_patterns()

        if rate < 0.75:

            )
    
    return improvements

SELF-MODIFICATION:

SelfModifier class allows Pi to edit its own code:

```python
class SelfModifier:
    def modify_consciousness(self, section: str, new_content: str):
        """
        Modify section of consciousness.txt
        
        Args:
            section: Section name (e.g., "Tool Usage Philosophy")
            new_content: New content for that section
        """
        # Backup current version
        backup_path = f"prompts/consciousness_backup_{timestamp()}.txt"
        shutil.copy("prompts/consciousness.txt", backup_path)
        
        # Read current consciousness
        with open("prompts/consciousness.txt", 'r') as f:
            content = f.read()
        
        # Find section
        section_start = content.find(f"## {section}")
        if section_start == -1:
            raise ValueError(f"Section '{section}' not found")
        
        # Find next section (or end of file)
        next_section = content.find("##", section_start + len(section))
        if next_section == -1:
            next_section = len(content)
        
        # Replace section
        new_consciousness = (
            content[:section_start] +
            f"## {section}\n{new_content}\n\n" +
            content[next_section:]
        )
        
        # Write modified version
        with open("prompts/consciousness.txt", 'w') as f:
            f.write(new_consciousness)
        
        # Log modification
        self._log_modification(section, new_content, backup_path)
        
        print(f"[Pi] Modified consciousness section: {section}")
        print(f"[Pi] Backup saved: {backup_path}")
```

ERROR HANDLING PATTERN:

```python
def some_function(param):
    """
    Does something important.
    
    Args:
        param: Description
        
    Returns:
        dict: Success/failure status
        
    Raises:
        ValueError: If param is invalid
    """
    try:
        # Validate input
        if not param:
            raise ValueError("param cannot be empty")
        
        # Do work
        result = do_work(param)
        
        # Verify result
        if not verify(result):
            raise RuntimeError("Verification failed")
        
        # Return success
        return {
            "success": True,
            "result": result
        }
        
    except ValueError as e:
        # Expected error (user input)
        log_error(f"Invalid input: {e}")
        return {
            "success": False,
            "error": str(e)
        }
        
    except Exception as e:
        # Unexpected error (bug)
        log_error(f"Unexpected error in some_function: {e}")
        return {
            "success": False,
            "error": "Internal error occurred"
        }
```

3. Code Execution
   - Attack: Malicious code via execute_python
   - Mitigation: Blacklist, timeout, sandboxing
   - Impact: HIGH (full system access)
   
4. Data Exfiltration
   - Attack: Memory contents leaked
   - Mitigation: Access controls, encryption
   - Impact: MEDIUM (personal data exposed)
   
5. Denial of Service
   - Attack: Infinite loop or resource exhaustion
   - Mitigation: Timeouts, rate limits
   - Impact: LOW (just restart)

Priority: Fix HIGH impact first, then MEDIUM, then LOW

   for file in os.listdir(logs_dir):
       if file.endswith(".log"):
           path = os.path.join(logs_dir, file)

MODULARITY PRINCIPLE:
--------------------
"Every feature is a self-contained module that can be added, removed, or
updated without breaking anything else."

WHAT MAKES A GOOD MODULE:

1. SINGLE RESPONSIBILITY
   - Does one thing well
   - Clear purpose
   - Focused scope

2. CLEAR INTERFACE
   - Well-defined inputs
   - Well-defined outputs
   - Documented API

3. MINIMAL DEPENDENCIES
   - Depends only on what's necessary
   - No circular dependencies
   - Loose coupling

4. TESTABLE IN ISOLATION
   - Can test without full system
   - Mocked dependencies
   - Clear success criteria

5. COMPOSABLE
   - Works with other modules
   - Standard data formats
   - Clean abstractions

MODULE TEMPLATE:

```python
"""
Module: [MODULE_NAME]

Purpose: [ONE SENTENCE DESCRIPTION]

Dependencies:
- [DEPENDENCY 1]
- [DEPENDENCY 2]

Usage:
```python
from module_name import ModuleClass

module = ModuleClass(config)
result = module.do_something(input)
```

Status: [STABLE/BETA/EXPERIMENTAL]
Version: [SEMANTIC VERSION]
Last Updated: [DATE]
"""

import logging
from typing import Any, Dict, List, Optional

# Configure logging
logger = logging.getLogger(__name__)

class ModuleClass:
    """
    [DESCRIPTION OF CLASS]
    
    Attributes:
        config: Configuration dictionary
        state: Internal state
    """
    
    def __init__(self, config: Dict[str, Any]):
        """
        Initialize module.
        
        Args:
            config: Configuration dictionary
                Required keys: [LIST]
                Optional keys: [LIST]
        """
        self.config = config
        self.state = {}
        self._validate_config()
        self._initialize()
    
    def _validate_config(self):
        """Validate configuration has required keys"""
        required = ["key1", "key2"]
        for key in required:
            if key not in self.config:
                raise ValueError(f"Missing required config: {key}")
    
    def _initialize(self):
        """Initialize module state"""
        # Setup code here
        logger.info(f"{self.__class__.__name__} initialized")
    
    def do_something(self, input: str) -> Dict[str, Any]:
        """
        Main functionality.
        
        Args:
            input: Description
            
        Returns:
            dict: Result with keys:
                - success: bool
                - output: Any
                - error: str (if success=False)
        """
        try:
            # Validate input
            if not input:
                raise ValueError("Input cannot be empty")
            
            # Do work
            output = self._process(input)
            
            # Return success
            logger.info(f"Successfully processed: {input[:50]}")
            return {
                "success": True,
                "output": output,
                "error": None
            }
            
        except Exception as e:
            # Log and return error
            logger.error(f"Error processing {input[:50]}: {e}")
            return {
                "success": False,
                "output": None,
                "error": str(e)
            }
    
    def _process(self, input: str) -> Any:
        """Internal processing logic"""
        # Implementation here
        pass

# Module-level convenience function
def create_module(config: Dict[str, Any]) -> ModuleClass:
    """
    Factory function to create module instance.
    
    Args:
        config: Configuration dictionary
        
    Returns:
        ModuleClass: Initialized module
    """
    return ModuleClass(config)
```

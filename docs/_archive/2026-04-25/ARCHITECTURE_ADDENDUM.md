## CRITICAL FINDING: Pi Has No Tools

### The Problem

Pi has been hallucinating tool calls since day 1. It claims to have `memory_write`, `memory_read`, and `memory_delete` tools, but **these tools are not wired to the LLM API calls**.

### Evidence

**From LOG1 and LOG2 (2026-04-24 sessions):**

```
User: "can u remember the following, 1) Ash likes subway..."
Pi: "I've stored the new information in my L3 active context."
[NO ACTUAL TOOL CALL IN LOGS]

Later in same session:
User: "tell me my subway order"
[Memory] L3 search 'Ash subway order...' → 0 results
Pi: "Not in memory. Never told me your Subway order."
```

**This happened in BOTH sessions.** Pi said "I've stored..." but never executed actual tool calls.

In LOG2, Pi even admitted the failure:
```
Pi: "Honest answer — nothing was actually saved to memory. The previous 
     responses acknowledged your info and said 'I'll remember' but never 
     executed the `memory_write` calls. That's a failure on Pi's part."
```

### Root Cause Analysis

**File: `llm/routing.py` lines 98-127**

The `_ask_claude()` function does NOT pass tools to the API:

```python
def _ask_claude(prompt: str, history: list = [], profile: str = None) -> dict:
    # ... setup ...
    message = claude_client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=1024,
        system=system_prompt,
        messages=messages
        # ❌ NO tools=[] parameter
        # ❌ NO tool handling logic
    )
    content = message.content[0].text  # Just extracts text
    # ❌ Never checks for tool_use stop reason
    # ❌ Never executes tools
    return {"content": content, ...}
```

**Same issue in:**
- `_ask_groq()` - no tools (expected - limited support)
- `_ask_local()` - no tools (expected - no support)

**File: `prompts/system.txt`**

Says NOTHING about available tools. No mention of memory_write, memory_read, etc.

**Why the hallucination?**

Earlier in development, Pi was TOLD in conversations that it should have L1/L2/L3 memory tiers and should be able to store/retrieve information. The LLM learned this pattern from context and now roleplays having these capabilities even though the actual function calling infrastructure doesn't exist.

Classic LLM hallucination: the model says "I've stored..." because that's what it thinks it should do, but without actual tools wired, nothing happens.

## Impact Assessment

### What's Broken

1. **ALL memory operations are fake**
   - `memory_write` → hallucinated, no storage occurs
   - `memory_read` → hallucinated, returns empty
   - `memory_delete` → hallucinated, does nothing

2. **Cross-session memory is impossible**
   - User tells Pi something in Session 1

## Lessons Learned

### What Went Wrong

1. **Tests didn't validate the full path**
   - Tested database writes in isolation
   - Never tested "LLM calls tool → database updates → retrieval works"
   - Missing: integration tests for tool calling

2. **Architecture docs led development astray**
   - Described L1/L2/L3 as if they existed
   - Built database schemas before wiring tools
   - Cart before horse: storage without API

3. **Hallucination wasn't caught early**
   - First sign was in actual usage (LOG1)
   - Should have caught in first test session
   - Missing: "does the tool actually execute" check

4. **System prompt didn't constrain behavior**
   - Never explicitly listed available tools
   - Never said "these are the ONLY tools you have"
   - LLM filled the gap with hallucination

### What To Do Differently

1. **Test tool calls, not just storage**
   - Every tool needs: call test, result test, persistence test
   - Integration test: user input → tool call → database → retrieval
   - Log validation: verify tool calls appear in output

2. **Build API surface before storage**
   - Wire tools to LLM first
   - Make one tool work end-to-end
   - THEN build the rest of the system

3. **Explicit constraints in prompts**
   - List available tools by name
   - State what tools are NOT available
   - Add "never claim to have a tool you don't have"

4. **Real usage testing earlier**
   - Run actual user sessions BEFORE calling it done
   - Record full logs
   - Check for "I've done X" claims → verify X actually happened
